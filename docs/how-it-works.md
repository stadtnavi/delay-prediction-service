# how `delay-prediction-service` works

*Note:* [*`delay-prediction-service` architecture*](architecture.md) provides a quick overview of the components this service consists of.

## 0. terminology

The terms used in the `delay-prediction-service` code base follow the [GTFS-Static](https://gtfs.org/reference/static/) & [GTFS-Realtime (GTFS-RT)](https://gtfs.org/reference/realtime/v2/) specs.

In addition, this project introduces the following terms:

- **run** – An "instance" of a GTFS *trip* on a single service day, e.g. trip `a-outbound-all-day-11` on service day `2021-06-15`.
- **vehicle position** – [WGS 84](https://epsg.io/4326) latitude & longitude; Usually includes a horizontal [dilution of precision](https://en.wikipedia.org/wiki/Dilution_of_precision_(navigation)) *hdop*.
- **vehicle position event** – A single vehicle position, including the date+time it has been observed at.
- **trajectory** – The spatio-temporal "progression" (a set of location & time pairs) of a *run*, defined by the GTFS-Static dataset.
- **match** – A *run*/*trajectory* determined to be the best "fitting" for a set of vehicle positions (see below).

## 2. fetch vehicle positions from Thingsboard

Currently, `delay-prediction-service` receives the *vehicle position events* from Thingsboard's WebSocket API (via [`thingsboard-telemetry-stream`](https://npmjs.com/package/thingsboard-telemetry-stream)).

Due to restarts or several service instances running, they may not be ordered, so we insert them into a `vehicle_positions` PostgreSQL table.

## 3. match against GTFS-Static runs

Whenever a *vehicle position event* has been received, the last 200 *vehicle position events* are read from the `vehicle_positions` PostgreSQL table. This spatio-temporal "progression" is compared with the *trajectories* of all currently "active" *runs*:

- For each pair (*vehicle position events*, *trajectory*), a score will be calculated. The lower the score, the better the "fit".
- The *trajectory* with the lowest score is considered to be the *match*, if
- it's score is low enough and sigificantly lower than the 2nd-lowest score (check the implementation).

This way, we only match a *trajectory* (and therefore a *run*) with the vehicle once we're reasonably sure.

## 4a. predict current delay, generate `TripUpdate`

Because the *trajectory* represents the planned equivalent to the vehicle's realtime movement, we can compute the delay by looking at when it should have been at its current position.

Using the delay, we can also determine which stop of the *run* is in the past and which is the next/upcoming one.

Using this data, we generate a [GTFS-RT `TripUpdate`](https://gtfs.org/reference/realtime/v2/#message-tripupdate).

## 4a. predict current position, generate `VehiclePosition`

Because the *trajectory* represents the planned equivalent to the vehicle's realtime movement, we can compute where the vehicle should be right now.

From this estimated position, we generate a [GTFS-RT `VehiclePosition`](https://gtfs.org/reference/realtime/v2/#message-vehicleposition).

## 5. publish predictions as GTFS-RT

We published the generated GTFS-RT messages via [MQTT](https://mqtt.org), to allow others to consume them in a streaming fashion.

In addition, we also generate a GTFS-RT "dump" and serve it via HTTP, to allow others to retrieve the data in bulk, and because this is the most standardized way to serve GTFS-RT data.
